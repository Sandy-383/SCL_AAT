PART 1: WHAT PROBLEM ARE WE SOLVING?
================================================================================

Sir/Ma'am, our project solves the "Transit Route Network Design Problem" (TRNDP).

In simple terms: given a city like Bangalore with 200 bus stops, we need to
find the BEST set of 10 bus routes such that:

  1. Passengers reach their destination FAST       (minimize travel time)
  2. Passengers don't need too many BUS CHANGES     (minimize transfers)
  3. The bus company spends LESS on operations       (minimize fleet cost)
  4. MAXIMUM people in the city are served            (maximize demand coverage)

These four goals CONFLICT with each other. For example:
  - If you want to cover every stop → you need more buses → cost goes UP
  - If you want fewer transfers → routes overlap more → waste of fleet

So there is NO single perfect answer. Instead, we find a SET of best trade-off
solutions called a "Pareto Front" — every solution on this front is optimal
in the sense that you can't improve one objective without worsening another.

This problem is NP-hard. For 200 stops and 10 routes, the search space is
astronomically large. Brute force is impossible. That's why we use a
nature-inspired optimization algorithm called Grey Wolf Optimizer (GWO),
and we accelerate it using NVIDIA GPU (CUDA programming).

PART 2: IS THIS "DISPLACEMENT"? — NO, THIS IS "OPTIMIZATION"
================================================================================

This project is NOT about displacement (like moving objects from point A to B
in physics). Let me clarify what it actually does:

WHAT IT IS:
  → It is a SEARCH/OPTIMIZATION problem.
  → We are searching through millions of possible route configurations
    to find ones that best satisfy our 4 objectives.
  → The "positions" in our code refer to MATHEMATICAL VECTORS in a
    220-dimensional search space — NOT physical locations.

THE WOLVES ARE METAPHORICAL:
  → In GWO, "wolves" are candidate solutions (possible route plans).
  → Each wolf's "position" is a vector of 220 numbers between 0 and 1.
  → These 220 numbers ENCODE a complete bus route plan:
      - Which stops each of the 10 routes visits
      - How frequently buses run on each route (headway)
      - How many buses are assigned to each route

THE "MOVEMENT" IS IN SEARCH SPACE:
  → When wolves "move" toward alpha/beta/delta, they are adjusting
    their 220 numbers to explore better route configurations.
  → It's like tuning 220 knobs simultaneously to find the sweet spot.
  → The GPU accelerates this "knob tuning" by computing all wolves
    and all dimensions in parallel.

ANALOGY FOR YOUR GUIDE:
  "Imagine 50 teams (wolves) each proposing a different bus system
  for Bangalore. The best 3 proposals (alpha, beta, delta) influence
  other teams to modify their proposals. After 50 rounds of revision,
  we get excellent bus route plans. The GPU lets all 50 teams work
  simultaneously instead of one at a time."

PART 3: HOW THE ALGORITHM WORKS — GREY WOLF OPTIMIZER (GWO)
================================================================================

GWO is inspired by how grey wolves hunt in packs. There is a strict
hierarchy:

  ALPHA (α)  = Best solution found so far        — Pack leader
  BETA  (β)  = Second best solution               — Second in command
  DELTA (δ)  = Third best solution                 — Third in command
  OMEGA (ω)  = All remaining wolves               — Followers

HUNTING MECHANISM (how solutions improve):

  Each wolf (candidate solution) updates its position by being "pulled"
  toward alpha, beta, and delta:

    Step 1: Calculate distance from current wolf to alpha
            D_alpha = |C1 × X_alpha − X_wolf|

    Step 2: Move toward alpha
            X1 = X_alpha − A1 × D_alpha

    Step 3: Repeat for beta → X2, and delta → X3

    Step 4: New position = average of three movements
            X_new = (X1 + X2 + X3) / 3

  Where:
    A = 2 × a × rand − a     (controls step size)
    C = 2 × rand              (controls how much leader influences)
    a = starts at 2, linearly decreases to 0 over iterations

  EARLY iterations (a close to 2):
    → A can be large → wolves EXPLORE far from leaders
    → This is the SEARCH phase — exploring diverse route configurations

  LATE iterations (a close to 0):
    → A is small → wolves CONVERGE toward leaders
    → This is the EXPLOITATION phase — refining the best routes found

MULTI-OBJECTIVE EXTENSION:
  Standard GWO finds a single best solution. We extended it to handle
  4 objectives simultaneously using:

    → PARETO ARCHIVE: We maintain a set of non-dominated solutions
      (solutions where no other solution is better in ALL objectives)

    → LEADER SELECTION FROM ARCHIVE: Alpha/beta/delta are picked from
      the Pareto archive using roulette wheel selection, biased toward
      LESS CROWDED regions (to maintain diversity)

    → NON-DOMINATED SORTING (NSGA-II style): After each iteration,
      we merge new solutions with the archive and keep only rank-0
      (non-dominated) solutions

    → CROWDING DISTANCE: Used to prune the archive when it exceeds
      100 solutions — we keep the most spread-out solutions

PART 4: HOW ROUTES ARE ENCODED — THE KEY DESIGN DECISION
================================================================================

The big challenge: GWO works in CONTINUOUS space (real numbers), but
bus routes are DISCRETE (specific stop sequences). How do we bridge this?

ENCODING SCHEME:

  Each route plan is represented as a vector x ∈ [0, 1]^220

  For 10 routes, each route gets 22 dimensions:
    - 20 dimensions for STOP SELECTION (which bus stops to visit)
    - 1 dimension for HEADWAY (how often the bus comes, in minutes)
    - 1 dimension for VEHICLE COUNT (how many buses on this route)

  Total: 10 routes × 22 dims = 220 dimensions

  DECODING (converting numbers back to actual route):
    - Stop index = round(x_i × 199)     [for 200 stops, maps 0→1 to 0→199]
    - Headway = 5 + x × (60 − 5)        [maps 0→1 to 5→60 minutes]
    - Vehicles = 1 + round(x × 14)      [maps 0→1 to 1→15 buses]

  EXAMPLE:
    If x = [0.15, 0.82, 0.03, 0.67, ...]
    Then:
      Stop 1 = round(0.15 × 199) = stop #30
      Stop 2 = round(0.82 × 199) = stop #163
      Stop 3 = round(0.03 × 199) = stop #6
      Stop 4 = round(0.67 × 199) = stop #133
      ...

  After decoding, DUPLICATE stops are removed and minimum route length
  (3 stops) is enforced.

WHY THIS ENCODING?
  → GWO naturally operates on continuous [0,1] vectors
  → GPU can compute position updates on all 220 dimensions simultaneously
  → No special discrete constraint handling needed
  → The continuous-to-discrete mapping is done ONLY during fitness evaluation

PART 5: FITNESS EVALUATION — HOW WE SCORE EACH ROUTE PLAN
================================================================================

For each wolf (candidate route plan), we compute 4 objective values:

─── OBJECTIVE f1: Average Travel Time ─────────────────────────────────────────

  We have a pre-computed travel time matrix T[i][j] = minutes from stop i to j

  For each pair of stops (i, j) that are served by the routes:
    weighted_sum += T[i][j] × demand[i][j]
    demand_sum   += demand[i][j]

  f1 = weighted_sum / demand_sum

  This gives the demand-weighted average travel time in minutes.
  Meaning: routes serving high-demand corridors count more.

─── OBJECTIVE f2: Transfer Ratio ──────────────────────────────────────────────

  For each pair of served stops (i, j):
    Check if ANY single route serves BOTH i and j
    If not → they need a TRANSFER

  f2 = (number of pairs needing transfer) / (total served pairs)

  Lower = better. Means more direct connectivity.

─── OBJECTIVE f3: Fleet Cost ──────────────────────────────────────────────────

  For each route:
    length_km  = total distance of the route (Haversine formula)
    fuel_cost  = ₹8/km × length_km × num_vehicles
    driver_cost = ₹150/hour × (length_km / 25 km/h) × num_vehicles
    route_cost = (fuel + driver) × 18 operating hours

  f3 = total cost across all routes (in ₹ thousands/day)

─── OBJECTIVE f4: Coverage Gap ────────────────────────────────────────────────

  served_stops = set of all stops appearing in any route
  covered_demand = total OD demand between served stop pairs
  total_demand = sum of entire demand matrix

  f4 = 1 − (covered_demand / total_demand)

  f4 = 0 means 100% coverage. f4 = 0.3 means 30% of demand is unserved.

─── CONSTRAINT PENALTIES ──────────────────────────────────────────────────────

  5 types of penalties (weight = 1000 each) are ADDED to all objectives
  if the solution violates constraints:

    1. ROUTE LENGTH: routes must have 3-20 stops
    2. CAPACITY: route capacity ≥ demand on that route
       capacity = vehicles × 60 passengers × (60/headway) trips/hour
    3. HEADWAY: must be between 5 and 60 minutes
    4. STOP COVERAGE: ≥ 80% of all stops must be served by at least one route
    5. ROUTE OVERLAP: if > 40% of edges are shared between routes → penalty

  The penalty makes infeasible solutions score very poorly, so the optimizer
  naturally moves AWAY from them without us explicitly rejecting solutions.

PART 6: WHERE DOES GPU (CUDA) COME IN?
================================================================================

The GPU accelerates the POSITION UPDATE step of each iteration.

─── WITHOUT GPU (CPU version): ────────────────────────────────────────────────

  for wolf = 0 to 49:               ← sequential, one at a time
    for dim = 0 to 219:             ← sequential, one at a time
      compute A1, C1, D_alpha
      compute X1 from alpha
      compute X2 from beta
      compute X3 from delta
      X_new[wolf][dim] = (X1 + X2 + X3) / 3

  Total operations: 50 × 220 = 11,000 sequential steps

─── WITH GPU (CUDA version): ──────────────────────────────────────────────────

  Launch kernel with grid = (50, 1) blocks, 256 threads per block
  EACH THREAD handles ONE wolf × ONE dimension simultaneously

  Thread (wolf=0, dim=0)  ──→  computes X_new[0][0]     ⎫
  Thread (wolf=0, dim=1)  ──→  computes X_new[0][1]     ⎬ ALL AT THE SAME TIME
  Thread (wolf=1, dim=0)  ──→  computes X_new[1][0]     ⎪ ON 2,560 CUDA CORES
  ...                                                     ⎪
  Thread (wolf=49, dim=219) → computes X_new[49][219]   ⎭

  11,000 operations done in ~1 step instead of 11,000 sequential steps!

─── GPU OPTIMIZATIONS IN OUR KERNEL: ──────────────────────────────────────────

  1. SHARED MEMORY: Alpha, beta, delta positions are loaded into fast
     on-chip shared memory (48 KB) so all threads in a block can read
     them without hitting slow global memory repeatedly.

  2. PRE-GENERATED RANDOM NUMBERS: Instead of generating randoms inside
     the kernel (expensive), we generate all 6 random values per wolf
     per dimension beforehand using CuPy and pass them as a flat array.

  3. NVRTC COMPILATION: The CUDA C kernel source code is compiled at
     runtime using NVIDIA Runtime Compilation — no need to install the
     full CUDA Toolkit. Just "pip install cupy-cuda12x" and it works.

  4. COALESCED MEMORY ACCESS: Consecutive threads access consecutive
     memory locations, maximizing GPU memory bandwidth utilization.

─── WHY FITNESS EVALUATION IS STILL ON CPU: ───────────────────────────────────

  The fitness function requires:
  - Graph traversal (Dijkstra shortest paths)
  - Set operations (finding which stops share routes)
  - Dynamic data structures (route decoding with duplicate removal)

  These are inherently sequential/branching operations that GPUs are
  BAD at. So we use a HYBRID approach:

    GPU  →  position updates (massively parallel, no branching)
    CPU  →  fitness evaluation (complex logic, graph algorithms)

  Data flows: GPU→CPU (download positions) then CPU→GPU (next iteration)

PART 7: EXECUTION FLOW — STEP BY STEP
================================================================================

When you type: python main.py --quick

  1. gpu_setup.py runs first
     → Adds NVIDIA DLL directories to Python's search path
     → Ensures CuPy can find CUDA libraries on Windows

  2. main.py parses arguments
     → --quick sets: 50 wolves, 50 iterations (instead of 100/500)

  3. CITY GENERATION (data/gtfs_loader.py)
     → Checks for GTFS CSV files in data/gtfs/ → none found
     → Falls back to SyntheticCityGenerator:
         • 40 stops in CBD (central Bangalore, inner 3 km)
         • 40 stops each in North, South, East, West zones
         • Generates Haversine-distance adjacency matrix [200×200]
         • Floyd-Warshall all-pairs shortest paths
         • Gravity-model OD demand matrix [200×200]
         • 10 initial greedy routes (residential→CBD)

  4. GRAPH CONSTRUCTION (data/graph_builder.py)
     → Creates NetworkX directed graph
     → Nodes = 200 stops (with lat, lon, zone, demand)
     → Edges = route connections (with travel time, distance)
     → Computes coverage ratio and connected components

  5. ENCODER SETUP (optimization/problem_encoder.py)
     → Dimension = 10 routes × (20 stops + 2) = 220
     → Maps [0,1]^220 ↔ route plans

  6. FITNESS SETUP (optimization/multi_objective.py)
     → Pre-computes base travel time matrix using Dijkstra [200×200]
     → This matrix is REUSED every iteration (not recomputed)

  7. ALGORITHM EXECUTION (for each of 5 algorithms):

     ┌─────────────────────────────────────────────────────────────┐
     │  CUDA-GWO (cuda_gwo/cupy_gwo.py)                          │
     │                                                             │
     │  a) CuPy compiles CUDA kernel via NVRTC        (~1 second) │
     │  b) Upload adj_matrix and demand to GPU memory              │
     │  c) Generate random initial population [50×220]             │
     │  d) Upload to GPU as float32 array                          │
     │  e) Evaluate initial population on CPU (50 fitness calls)   │
     │  f) Initialize Pareto archive                               │
     │                                                             │
     │  FOR iteration = 0 to 49:                                   │
     │    1. Select α, β, δ from archive (roulette wheel)          │
     │    2. Upload leaders to GPU                                 │
     │    3. Generate random matrix [50×6×220] on GPU              │
     │    4. LAUNCH KERNEL: gwo_update_positions                   │
     │       → 50 blocks × 1 dim-chunk × 256 threads              │
     │       → Shared memory: 3 × 256 × 4 bytes = 3 KB            │
     │       → Each thread: read 6 randoms, compute X1+X2+X3/3    │
     │    5. Synchronize GPU (wait for kernel to finish)           │
     │    6. Download updated positions to CPU                     │
     │    7. CPU: decode 50 positions → 50 route plans             │
     │    8. CPU: evaluate 50 route plans (4 objectives each)      │
     │    9. Update Pareto archive (non-dominated sort + prune)    │
     │   10. Log: HV, best_f1, iteration time                     │
     │                                                             │
     │  g) Return best solution (min f1 from archive)              │
     └─────────────────────────────────────────────────────────────┘

     Then CPU-GWO, PSO, GA, ACO run similarly but ALL on CPU.

  8. VISUALIZATION (visualization/)
     → route_map.html: Interactive Folium map of Bangalore
       (each route is a colored polyline, stops are circle markers)
     → dashboard.png: 6-panel summary figure
     → Convergence plots, Pareto front, speedup chart

  9. QUALITY EVALUATION (experiments/solution_quality.py)
     → Computes Hypervolume, GD, IGD, Spacing for each algorithm
     → Saves results_table.tex (paste directly into LaTeX paper)
     → Saves solution_quality.json (raw metrics)

  10. FINAL SUMMARY printed to console:
      cuda_gwo    f1=259.17 min  f2=138.32 tx  f3=4843.4 kRs  time=43s
      cpu_gwo     f1=259.17 min  f2=138.32 tx  f3=4843.4 kRs  time=43s
      ...

PART 8: COMPARISON WITH BASELINE ALGORITHMS
================================================================================

We compare CUDA-GWO against 4 established metaheuristics:

┌────────────────┬──────────────────────────────────────────────────────────┐
│ Algorithm      │ How it works                                            │
├────────────────┼──────────────────────────────────────────────────────────┤
│ CPU-GWO        │ Same as CUDA-GWO but runs sequentially on CPU.          │
│                │ Used to measure GPU speedup.                            │
├────────────────┼──────────────────────────────────────────────────────────┤
│ PSO            │ Particles have velocity. Each particle moves toward     │
│                │ its personal best AND the global best (from archive).    │
│                │ velocity = w×vel + c1×r1×(pbest−pos) + c2×r2×(gbest−pos)│
├────────────────┼──────────────────────────────────────────────────────────┤
│ GA (NSGA-II)   │ Tournament selection → SBX crossover → polynomial       │
│                │ mutation. Environmental selection by Pareto rank +       │
│                │ crowding distance. Most established multi-obj method.    │
├────────────────┼──────────────────────────────────────────────────────────┤
│ ACO            │ Ants deposit "pheromone" on good positions.              │
│                │ 220-dim space discretized into 20 bins per dimension.    │
│                │ Each ant samples bins proportional to pheromone level.   │
│                │ Good solutions reinforce their bin's pheromone.          │
└────────────────┴──────────────────────────────────────────────────────────┘

QUALITY METRICS WE REPORT:

  HV  (Hypervolume)  ↑ higher is better
    → Volume of objective space dominated by our Pareto front.
    → Measures both convergence AND diversity simultaneously.
    → For 4 objectives, computed via Monte Carlo (50,000 samples).

  GD  (Generational Distance)  ↓ lower is better
    → Average distance from our front to the "true" Pareto front.
    → True front = union of all algorithms' non-dominated solutions.

  IGD (Inverted Generational Distance)  ↓ lower is better
    → Average distance from true front to our front.
    → Penalizes missing regions of the true front.

  Spacing  ↓ lower is better
    → Std. deviation of nearest-neighbor distances on our front.
    → Measures how uniformly spread our solutions are.

PART 11: POTENTIAL QUESTIONS YOUR GUIDE MIGHT ASK
================================================================================

Q: "Why GWO and not Genetic Algorithm or PSO?"
A: GWO has fewer hyperparameters (just 'a' which decreases linearly).
   GA needs crossover rate, mutation rate, selection pressure.
   PSO needs inertia, cognitive/social coefficients.
   GWO's simplicity makes it easier to parallelize on GPU.
   Also, GWO's exploration-exploitation balance is naturally controlled
   by the linear decrease of parameter 'a' from 2 to 0.

Q: "How much speedup does GPU give?"
A: For the position update step alone: significant parallelism.
   However, the overall speedup is limited by the CPU fitness evaluation
   (Amdahl's Law). The real benefit shows at larger problem sizes
   (500-1000 stops) where the 220→2200 dimensions make the sequential
   position update the bottleneck.

Q: "Why not evaluate fitness on GPU too?"
A: Fitness requires graph traversal (shortest paths), set operations
   (finding transfers), and dynamic data structures (variable-length
   routes). These involve heavy branching which GPUs handle poorly.
   A fully GPU fitness evaluation would require implementing a GPU
   Dijkstra algorithm and GPU-resident graph, which is a research
   project in itself.

Q: "What is the Pareto front?"
A: A set of solutions where none is strictly better than any other.
   Solution A dominates B if A is ≤ B in ALL objectives and < B in
   at least one. The Pareto front = all non-dominated solutions.
   Example: Route plan X has f1=200min, f3=3000Rs.
            Route plan Y has f1=250min, f3=2000Rs.
   Neither dominates the other → both are on the Pareto front.
   The decision-maker picks based on their priority.

Q: "How is the city generated?"
A: We use Bangalore coordinates (12.97°N, 77.59°E) as center.
   40 stops placed randomly within 3km (CBD zone),
   160 stops across 4 radial zones (N/S/E/W, 3-15km from center).
   Travel times: Haversine distance / 25 km/h × 1.3 (congestion).
   All-pairs shortest paths via Floyd-Warshall O(n³).
   Demand: gravity model — proportional to √(demand_i × demand_j).

Q: "Can this work with real city data?"
A: Yes! The --gtfs flag accepts real GTFS (General Transit Feed
   Specification) data — the standard format used by Google Maps.
   If you provide stops.txt, routes.txt, trips.txt, stop_times.txt,
   the system will use real stop locations and route information.
